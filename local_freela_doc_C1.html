1. Visão Geral do Projeto

A Local Freela é uma plataforma web que conecta pessoas que oferecem serviços (prestadores) a pessoas que os buscam (solicitantes). Usuários criam perfis, anunciam suas habilidades e negociam a prestação de serviços de forma direta, com avaliação ao final. A plataforma é exclusivamente web, com possibilidade de expansão futura para aplicativo mobile.

1.1 Definição do Minimundo

A Local Freela é uma plataforma web de intermediação de serviços que conecta pessoas que oferecem habilidades — denominadas prestadores — a pessoas que necessitam contratar um serviço — denominadas solicitantes. A plataforma opera exclusivamente via navegador web e é voltada ao público geral, independentemente de área de atuação ou formação acadêmica.

Qualquer pessoa pode se cadastrar na plataforma informando nome, e-mail, senha, uma breve biografia e foto de perfil. Após a autenticação, o usuário pode exercer ambos os papéis simultaneamente: atuar como prestador, ao anunciar suas habilidades e atender pedidos de serviço, e como solicitante, ao buscar serviços e enviar requisições a outros prestadores.

O prestador cria anúncios de habilidades informando título, descrição, categoria temática e preço fixo pelo serviço. Os anúncios são exibidos publicamente na plataforma e podem ser encontrados por meio de busca dinâmica por palavra-chave ou filtragem por categoria, sem necessidade de recarregar a página.

Ao encontrar um anúncio de interesse, o solicitante envia uma solicitação de serviço acompanhada de mensagem descritiva. Essa ação consome tokens do saldo do solicitante — moeda interna da plataforma —, desincentivando solicitações em massa e garantindo a qualidade dos contatos recebidos. O prestador é imediatamente notificado dentro da própria plataforma, sem envio de e-mail ou notificação externa, e pode aceitar ou rejeitar a solicitação. Solicitações não respondidas dentro do prazo são automaticamente marcadas como expiradas pelo sistema.

Com a solicitação aceita, o serviço entra em andamento. Ao ser concluído, qualquer uma das partes registra a conclusão. O solicitante, então, pode registrar uma avaliação com nota de 1 a 5 estrelas e comentário opcional, contribuindo para a formação da reputação pública do prestador.

Os tokens são a moeda interna da Local Freela: ganhos ao realizar ações positivas (concluir serviços, receber boas avaliações, completar perfil, indicar novos usuários) e gastos para interagir com a plataforma (enviar solicitações, enviar mensagens, destacar perfil). Toda movimentação é registrada no histórico individual. Semanalmente, a plataforma publica um ranking público dos prestadores e contratantes mais ativos, acessível sem login. A administração é responsável pelo gerenciamento de usuários e pelo cadastro das categorias de habilidades.

4. Sistema de Tokens

🪙 Observação da Equipe: A equipe optou por implementar um sistema proprietário de tokens como principal diferencial competitivo da Local Freela em relação a outras soluções similares no mercado. Os tokens criam uma economia interna que agrega valor, combate spam e incentiva comportamentos positivos.

4.1 Créditos para Solicitação de Serviços

🔴 Ações de Gasto
AçãoCusto
Enviar solicitação de serviço🪙 2 tokens
Enviar mensagem direta ao prestador🪙 1 token
Destacar anúncio de habilidade🪙 5 tokens
Destacar perfil no ranking🪙 5 tokens
Subir posição no ranking🪙 3 tokens
🟢 Ações de Ganho
AtividadeGanho
Completar o cadastro do perfil🪙 +3 tokens
Concluir um serviço com sucesso🪙 +5 tokens
Receber avaliação positiva (≥ 4 estrelas)🪙 +2 tokens
Indicar novo usuário que se cadastra🪙 +4 tokens

4.2 Ranking da Plataforma

Categoria do RankingCritérioBenefício
🏆 Top Prestadores da SemanaMaior volume de tokens recebidos por avaliações e conclusõesDestaque na página inicial
🏢 Top Contratantes da SemanaMaior volume de tokens gastos em solicitações aceitasBadge "Parceiro Local Freela"

4.3 Requisitos Funcionais – Tokens (RF13 a RF15)

IDDescriçãoPrioridade
RF13O sistema deve manter saldo de tokens por usuário, debitando automaticamente o custo de cada ação e validando saldo suficiente antes de prosseguir.Alta
RF14O sistema deve creditar tokens automaticamente ao usuário ao detectar atividades recompensáveis. Cada transação deve ser registrada com motivo, quantidade e data.Alta
RF15O sistema deve calcular e exibir ranking semanal público de prestadores e contratantes baseado na movimentação de tokens, acessível sem login.Média

5. Requisitos Não Funcionais

Os RNFs definem atributos de qualidade do sistema. O RNF06 especifica o comportamento do sistema de notificações, alinhado ao RF12.
IDCategoriaDescrição
RNF01SegurançaA autenticação deve utilizar mecanismo seguro com expiração de sessão configurável, validado em todas as rotas protegidas.
RNF02SegurançaSenhas dos usuários devem ser armazenadas exclusivamente como hash. Nenhuma senha em texto plano deve trafegar ou ser persistida.
RNF03DesempenhoO tempo de resposta para operações de busca e filtro de habilidades deve ser inferior a 2 segundos em condições normais de uso.
RNF04UsabilidadeA interface deve ser responsiva, adaptando-se a dispositivos desktop e mobile, seguindo princípios de acessibilidade básica.
RNF05ManutenibilidadeO código deve ser modular e documentado, seguindo boas práticas (clean code, separação de responsabilidades).
RNF06NotificaçõesAs notificações do RF12 devem ser exclusivamente do tipo in-app. O sistema não deve depender de serviços externos de e-mail, push ou SMS.
RNF07TokensTodas as transações de tokens devem ser atômicas (SQLite com suporte a transações via BEGIN/COMMIT), garantindo consistência do saldo mesmo em falha parcial. Saldo nunca negativo.
RNF08CompatibilidadeO sistema deve funcionar corretamente nos navegadores modernos: Chrome, Firefox, Edge e Safari (versões atuais e anterior).
RNF09ArquiteturaO backend deve ser stateless, seguindo o estilo arquitetural cliente-servidor com padrão MVC, retornando respostas em JSON.

9. DDL – Criação de Tabelas com SQL

Scripts de criação do banco de dados para SQLite 3. A ordem de criação respeita as dependências de chaves estrangeiras. Para ativar o suporte a FK no SQLite, execute PRAGMA foreign_keys = ON; ao abrir a conexão.
1 · Banco de dados
CREATE DATABASE IF NOT EXISTS localfreela DEFAULT CHARACTER SET utf8mb4 DEFAULT COLLATE utf8mb4_unicode_ci; USE localfreela;
2 · Tabela CATEGORIA
CREATE TABLE categoria ( id BIGINT NOT NULL AUTO_INCREMENT, nome VARCHAR(100) NOT NULL, icone VARCHAR(50), PRIMARY KEY (id) ) ENGINE=InnoDB;
3 · Tabela USUARIO
CREATE TABLE usuario ( id BIGINT NOT NULL AUTO_INCREMENT, nome VARCHAR(150) NOT NULL, email VARCHAR(150) NOT NULL, senha_hash VARCHAR(255) NOT NULL, bio TEXT, foto_perfil VARCHAR(255), reputacao DECIMAL(3,2) NOT NULL DEFAULT 0.00, saldo_tokens INT NOT NULL DEFAULT 0, data_cadastro DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP, PRIMARY KEY (id), UNIQUE KEY uq_email (email) ) ENGINE=InnoDB;
4 · Tabela HABILIDADE
CREATE TABLE habilidade ( id BIGINT NOT NULL AUTO_INCREMENT, titulo VARCHAR(150) NOT NULL, descricao TEXT, preco DECIMAL(10,2) NOT NULL, disponivel TINYINT(1) NOT NULL DEFAULT 1, data_criacao DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP, usuario_id BIGINT NOT NULL, categoria_id BIGINT NOT NULL, PRIMARY KEY (id), CONSTRAINT fk_hab_usuario FOREIGN KEY (usuario_id) REFERENCES usuario(id), CONSTRAINT fk_hab_categoria FOREIGN KEY (categoria_id) REFERENCES categoria(id) ) ENGINE=InnoDB;
5 · Tabela SOLICITACAO
CREATE TABLE solicitacao ( id BIGINT NOT NULL AUTO_INCREMENT, mensagem TEXT, status ENUM('PENDENTE', 'ACEITA', 'CONCLUIDA', 'EXPIRADA', 'REJEITADA') NOT NULL DEFAULT 'PENDENTE', data_envio DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP, data_expiracao DATETIME, data_atualizacao DATETIME ON UPDATE CURRENT_TIMESTAMP, solicitante_id BIGINT NOT NULL, habilidade_id BIGINT NOT NULL, prestador_id BIGINT NOT NULL, PRIMARY KEY (id), CONSTRAINT fk_sol_solicitante FOREIGN KEY (solicitante_id) REFERENCES usuario(id), CONSTRAINT fk_sol_habilidade FOREIGN KEY (habilidade_id) REFERENCES habilidade(id), CONSTRAINT fk_sol_prestador FOREIGN KEY (prestador_id) REFERENCES usuario(id) ) ENGINE=InnoDB;
6 · Tabela AVALIACAO
CREATE TABLE avaliacao ( id BIGINT NOT NULL AUTO_INCREMENT, nota TINYINT NOT NULL, comentario TEXT, data DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP, solicitacao_id BIGINT NOT NULL, avaliador_id BIGINT NOT NULL, PRIMARY KEY (id), UNIQUE KEY uq_aval_sol (solicitacao_id), CONSTRAINT ck_nota CHECK (nota BETWEEN 1 AND 5), CONSTRAINT fk_aval_sol FOREIGN KEY (solicitacao_id) REFERENCES solicitacao(id), CONSTRAINT fk_aval_avaliador FOREIGN KEY (avaliador_id) REFERENCES usuario(id) ) ENGINE=InnoDB;
7 · Tabela NOTIFICACAO
CREATE TABLE notificacao ( id BIGINT NOT NULL AUTO_INCREMENT, mensagem TEXT NOT NULL, lida TINYINT(1) NOT NULL DEFAULT 0, tipo VARCHAR(20) NOT NULL DEFAULT 'INTERNA', data_criacao DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP, usuario_id BIGINT NOT NULL, PRIMARY KEY (id), CONSTRAINT fk_noti_usuario FOREIGN KEY (usuario_id) REFERENCES usuario(id) ) ENGINE=InnoDB;
8 · Tabela TRANSACAO_TOKEN
CREATE TABLE transacao_token ( id BIGINT NOT NULL AUTO_INCREMENT, tipo ENUM('GANHO', 'GASTO') NOT NULL, quantidade INT NOT NULL, motivo VARCHAR(255), data DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP, usuario_id BIGINT NOT NULL, PRIMARY KEY (id), CONSTRAINT fk_token_usuario FOREIGN KEY (usuario_id) REFERENCES usuario(id) ) ENGINE=InnoDB;
✔ Ordem de criação: categoria → usuario → habilidade → solicitacao → avaliacao → notificacao → transacao_token. Esta sequência garante que todas as FK já referenciem tabelas existentes.

12.3 Modelagem do Banco de Dados

O MER completo está documentado na Seção 8 (Figura 3) e o DDL na Seção 9. O banco de dados utilizado é o SQLite 3, embutido no Python sem necessidade de servidor externo.
